# 代码审查流程 (Code Review Process)

## 目的

代码审查确保：
- 代码质量和可维护性
- 知识共享
- 减少Bug
- 统一代码风格

## 审查清单

### 功能性
- [ ] 代码实现了需求
- [ ] 边界条件处理正确
- [ ] 错误处理完善
- [ ] 没有明显的Bug

### 代码质量
- [ ] 代码清晰易读
- [ ] 命名规范合理
- [ ] 函数职责单一
- [ ] 避免重复代码
- [ ] 注释清晰必要

### 测试
- [ ] 包含单元测试
- [ ] 测试覆盖率达标
- [ ] 测试用例充分
- [ ] 测试通过

### 性能
- [ ] 没有性能问题
- [ ] 数据库查询优化
- [ ] 避免N+1查询
- [ ] 合理使用缓存

### 安全
- [ ] 输入验证
- [ ] SQL注入防护
- [ ] XSS防护
- [ ] 敏感信息保护

## 审查流程

1. **自审**: 提交前自己先审查一遍
2. **提交PR**: 填写完整的PR描述
3. **自动检查**: CI/CD自动运行测试
4. **人工审查**: 至少1人审查通过
5. **修改**: 根据反馈修改代码
6. **合并**: 审查通过后合并

## 审查时间

- 小型PR（<200行）: 4小时内
- 中型PR（200-500行）: 1个工作日内
- 大型PR（>500行）: 2个工作日内

## 审查意见

使用以下标签：
- **MUST**: 必须修改
- **SHOULD**: 建议修改
- **NIT**: 小问题，可选
- **QUESTION**: 疑问

## 最佳实践

### 对于作者
- 保持PR小而专注
- 提供清晰的描述
- 及时响应反馈
- 保持友好态度

### 对于审查者
- 及时审查
- 提供建设性意见
- 解释原因
- 保持尊重
