## 1. 产品定位

### 1.1 产品简介

**智能售后工作台**是一款智能化的客户服务平台，通过AI辅助技术为客服团队提供全链路支持——从对话管理、客户档案、知识检索到质量检验和任务协同，帮助企业提升客服效率和服务质量。

**核心Slogan**: "让AI辅助每个客服，提升效率与质量"

**产品特点**:
- ✅ **统一工作台**: 多渠道对话聚合（飞书、企微、Web）
- ✅ **AI智能辅助**: 情感分析、响应建议、智能质检
- ✅ **知识高效管理**: 结构化知识库，快速检索与引用
- ✅ **全流程协同**: 对话→任务→质检→报表闭环
- ✅ **数据驱动决策**: 实时报表和质量洞察

---

### 1.2 市场定位

**目标市场**: 中国B2B/B2C企业客户服务场景

**目标客户**:
- 智能硬件公司售后团队（10-50人）
- SaaS企业客户成功团队（5-30人）
- 电商平台客服中心（20-100人）
- 教育培训机构咨询团队（5-20人）

**竞争定位**:
| 维度 | 传统客服系统 | 智能客服机器人 | **智能售后工作台** |
|-----|------------|--------------|------------------|
| **核心价值** | 工单管理 | 自动应答 | **AI辅助人工** |
| **AI能力** | 无 | 单一问答Bot | **多场景AI辅助** |
| **知识管理** | 人工维护 | 预设FAQ | **结构化知识库+推荐** |
| **质检能力** | 手动抽检 | 无 | **AI全量质检** |
| **适用场景** | 基础工单 | 标准化咨询 | **复杂售后问题** |

**差异化优势**:
1. ✅ **AI辅助而非替代**: 保留人工判断灵活性，AI提供决策支持
2. ✅ **全流程覆盖**: 对话→客户→知识→任务→质检全链路管理
3. ✅ **轻量化部署**: Web应用，无需复杂安装配置
4. ✅ **数据洞察**: 实时质检和报表，持续改进服务质量

## 2. 目标用户

### 2.1 核心用户画像

#### 用户1: 一线客服人员 (Primary User)

**人物角色**: 李小美，26岁，某智能硬件公司售后客服

**工作场景**:
- 每天处理20-40个客户咨询（产品使用、故障排查、退换货）
- 需要快速查找知识库答案
- 需要记录客户信息和服务历史
- 需要创建跟进任务（回电、升级、交付）
- 每周接受质检考核

**核心痛点**:
- ❌ 知识库检索慢，找不到答案浪费时间
- ❌ 客户信息分散，难以快速了解历史
- ❌ 任务容易遗漏，影响客户满意度
- ❌ 质检标准不清晰，被扣分不知道如何改进

**期望价值**:
- ✅ 快速检索知识，一键引用到回复
- ✅ 完整客户档案，了解历史互动
- ✅ 任务自动提醒，不遗漏任何跟进
- ✅ 质检结果透明，有具体改进建议

**使用频率**: 每工作日 6-8 小时

---

#### 用户2: 质检专员 (Power User)

**人物角色**: 王经理，32岁，客服团队质检主管

**工作场景**:
- 每天抽检30-50条对话，检查服务质量
- 需要生成周报/月报给管理层
- 发现问题后需要跟进改进
- 需要分析质量趋势，发现系统性问题

**核心痛点**:
- ❌ 手动质检效率低，覆盖率不足10%
- ❌ 评分标准主观性强，容易引发争议
- ❌ 报表制作耗时，数据分析不深入
- ❌ 改进措施难以落地和追踪

**期望价值**:
- ✅ AI辅助质检，提升覆盖率和效率
- ✅ 量化评分标准，减少主观争议
- ✅ 自动生成报告，数据可视化
- ✅ 针对性改进建议，可追踪效果

**使用频率**: 每工作日 4-6 小时

---

#### 用户3: 团队经理 (Decision Maker)

**人物角色**: 张总，38岁，客服中心负责人

**工作场景**:
- 管理20人客服团队
- 关注团队效率和客户满意度
- 每月向上级汇报服务质量数据
- 需要识别问题并优化流程

**核心痛点**:
- ❌ 团队效率不透明，无法量化评估
- ❌ 客户满意度波动，不知道根因
- ❌ 缺乏数据支持，决策靠经验
- ❌ 问题发现滞后，难以及时干预

**期望价值**:
- ✅ 实时数据看板，掌握团队状态
- ✅ 质量趋势分析，发现问题根因
- ✅ 数据驱动决策，优化资源配置
- ✅ 早期预警机制，及时干预问题

**使用频率**: 每工作日 1-2 小时

#### 4.2.2 核心聚合根设计

系统实现了5个核心聚合根，采用DDD战术设计模式：

**聚合根1: Conversation (对话聚合根)**
- **代码位置**: `backend/src/domain/conversation/models/Conversation.ts`
- **核心值对象**: Channel, AgentMode, ConversationStatus, MessagePriority

**聚合根2: CustomerProfile (客户画像聚合根)**
- **代码位置**: `backend/src/domain/customer/models/CustomerProfile.ts`
- **核心职责**: 管理客户360°画像、健康度评分、风险评估
- **领域服务**: HealthScoreCalculator, RiskEvaluator

**聚合根3: Task (任务聚合根)**
- **代码位置**: `backend/src/domain/task/models/Task.ts`
- **核心职责**: 任务流转、质检达标验证、超期告警
- **核心值对象**: TaskPriority, TaskStatus, QualityScore
- **领域方法**: isOverdue(), needsEscalation(), canComplete() (质检>=70分)

**聚合根4: Requirement (需求聚合根)**
- **代码位置**: `backend/src/domain/requirement/models/Requirement.ts`
- **核心职责**: AI需求检测、优先级评估、自动创建工单
- **核心值对象**: Priority, RequirementSource
- **领域方法**: shouldAutoCreateTask(), needsCustomerCommunication()

**聚合根5: KnowledgeItem (知识库聚合根)**
- **代码位置**: `backend/src/domain/knowledge/models/KnowledgeItem.ts`
- **核心职责**: 知识管理、标签分类、使用统计
- **核心值对象**: KnowledgeCategory
- **领域方法**: 15个业务方法，包含isOutdated(), calculateRelevanceScore()等

**详细设计**: 参见《DDD技术架构设计文档 - 第三章战术设计》

### 4.3 技术栈选型

#### 4.3.1 前端技术栈

```javascript
{
  "framework": "Vanilla JavaScript ES2022+",
  "language": "JavaScript (ES Module)",
  "styling": "Tailwind CSS 3.4",
  "stateManagement": "自研Store模式 (轻量级)",
  "routing": "单页应用 (SPA)",
  "http": "Fetch API + 封装拦截器",
  "realtime": "原生WebSocket + 心跳重连",
  "charts": "ECharts 5.4 (报表看板)",
  "build": "Vite 5.0",
  "codeQuality": "ESLint + Prettier"
}
```

**选型理由**:
- **Vanilla JavaScript**: 性能优异，无虚拟DOM开销，包体积小(<100KB)
- **Tailwind CSS**: 原子化CSS，快速开发，高度可定制
- **自研Store**: 轻量级状态管理，类Redux模式，满足需求无需引入重型框架
- **原生WebSocket**: Agent通信实时性要求高，原生API性能最优
- **Vite**: 快速HMR，冷启动<3秒，开发体验好
- **模块化设计**: 组件化思想，目录结构清晰 (assets/js/modules/*, assets/css/components/*)

---

#### 4.3.2 后端技术栈

```typescript
{
  "framework": "Fastify 4.25",
  "language": "TypeScript 5.3",
  "orm": "TypeORM 0.3",
  "architecture": "DDD (Domain-Driven Design)",
  "database": {
    "relational": "PostgreSQL 15",
    "cache": "Redis 7.2",
    "vector": "Milvus 2.3",
    "storage": "MinIO"
  },
  "realtime": "原生WebSocket",
  "eventBus": "自研EventBus + Outbox模式",
  "validation": "Joi / Zod",
  "logging": "Winston + 结构化日志",
  "testing": "Jest + Supertest",
  "deployment": "Docker + Docker Compose"
}
```

**DDD分层架构**:
- **表现层**: REST API Routes + WebSocket Handlers
- **应用层**: UseCase编排 + SAGA协调器 + EventHandlers + 防腐层ACL
- **领域层**: 5个聚合根 + 领域服务 + 23+领域事件 + Specification模式(35+实现)
- **基础设施层**: Repository实现 + Outbox模式EventBus + 外部集成 (AgentScope/LLM API)

**选型理由**:
- **Fastify**: 高性能Node.js框架，吞吐量>3万req/s，schema验证内置
- **TypeORM**: 支持多种数据库，Active Record + Data Mapper模式，事务支持完善
- **PostgreSQL**: ACID保证，JSONB存储聚合根，全文检索能力强
- **Redis**: 高性能缓存，支持Pub/Sub，分布式锁
- **Milvus**: 专业向量数据库，HNSW索引，ms级检索
- **Outbox模式**: 事件可靠性100%，与聚合根同事务保存，后台异步发布
